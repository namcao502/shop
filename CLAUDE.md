## Environment

Copy `.env.local.example` to `.env.local`. Required variables:

- `NEXT_PUBLIC_FIREBASE_*` — Firebase client SDK (6 keys from Firebase console)
- `FIREBASE_ADMIN_PROJECT_ID`, `FIREBASE_ADMIN_CLIENT_EMAIL`, `FIREBASE_ADMIN_PRIVATE_KEY` — Firebase Admin SDK (service account)
- `VIETQR_BANK_ID`, `VIETQR_ACCOUNT_NUMBER`, `VIETQR_ACCOUNT_NAME` — VCB bank account for QR generation
- `MOMO_PARTNER_CODE`, `MOMO_ACCESS_KEY`, `MOMO_SECRET_KEY`, `MOMO_ENDPOINT` — MoMo payment gateway
- `NEXT_PUBLIC_BASE_URL` — base URL for MoMo webhook `returnUrl` (e.g. `http://localhost:3000`)

## Architecture

**Stack:** Next.js 16 App Router · React 19 · TypeScript · Tailwind CSS · Firebase 12 (Auth, Firestore, Storage) · Zod 4 · VietQR · MoMo

### Data flow rule: all writes go through API routes

Customer-facing reads (products, categories) hit Firestore directly from the client. Everything that mutates state — order creation, payment initiation, admin order updates — goes through Next.js API routes (`src/app/api/`) that use the Firebase **Admin SDK**. This is what lets the server enforce prices, validate stock, and sign HMAC webhooks without trusting the client.

**Exception — product image uploads:** Admins upload images directly to Firebase Storage from the client (`src/components/admin/ImageUploader.tsx`) using the client SDK. Files are stored at `products/{timestamp}_{i}_{filename}`. The download URL is then saved in the product's `images: string[]` field via the normal Firestore write path. This is safe because Firebase Storage rules restrict uploads to authenticated admins.

### Authentication layers

- **Client auth:** `AuthProvider` (`src/lib/firebase/auth-context.tsx`) wraps the app; `useAuth()` gives `user`, `signIn`, `signOut`, `getIdToken`
- **Server auth:** all API routes call `verifyAuth()` or `verifyAdminAuth()` from `src/lib/verify-admin.ts`, which verifies the Firebase ID token and checks `isAdmin` from Firestore
- **Admin access:** set `isAdmin: true` directly on the user document in Firestore (Firebase console or via a one-off script). The middleware (`src/proxy.ts`) matches `/admin/*` routes but actual auth is enforced in the admin layout client-side and in every admin API route server-side

### API routes

| Route | Methods | Purpose |
|-------|---------|---------|
| `/api/orders` | `POST` | Create order (atomic stock decrement + notification) |
| `/api/orders/[id]` | `PATCH` | Order actions: `cancel`, `update_address` (customer); `confirm_payment`, `ship`, `deliver` (admin) |
| `/api/vietqr` | `POST` | Returns VietQR image URL for a given order |
| `/api/momo/create` | `POST` | Initiates MoMo payment, returns redirect URL |
| `/api/momo/callback` | `POST` | MoMo webhook — verifies HMAC, updates payment status |
| `/api/notifications/read-all` | `POST` | Marks all notifications as read for the authenticated user |

### Order creation (the critical path)

`POST /api/orders` runs a Firestore **transaction** that reads stock, decrements it, creates the order, and writes an `order_placed` notification atomically — preventing overselling. After the order is created, the client calls either `/api/vietqr` (returns a QR image URL) or `/api/momo/create` (returns a MoMo redirect URL). MoMo sends a webhook to `/api/momo/callback`, which verifies the HMAC signature before updating payment status and writing a `payment_received` notification.

`orderCode` (e.g. `ORD0042`) is generated by atomically incrementing `counters/orders` in Firestore (`src/lib/order-code.ts`). It doubles as the VietQR transfer note for manual payment matching.

### Order actions (PATCH /api/orders/[id])

Supported `action` values in the request body:

- **Customer actions** (require ownership): `cancel`, `update_address`
- **Admin actions** (require `isAdmin`): `confirm_payment`, `ship`, `deliver`

Each action writes a corresponding notification via `writeNotification()`.

### Notifications

`src/lib/notifications.ts` exports `writeNotification(data, tx?)`:
- Pass a Firestore transaction `tx` to include the write atomically (used in order creation).
- Without `tx`, the write is fire-and-forget (used in MoMo callback).
- `userId` is the Firebase UID for user notifications or `"admin"` for admin-targeted ones.
- Stored in the `notifications` Firestore collection.
- `NotificationBell` (`src/components/layout/NotificationBell.tsx`) polls and displays unread counts in the header.

`NotificationType` covers: `order_placed`, `order_shipped`, `order_delivered`, `order_cancelled`, `payment_confirmed`, `address_updated`, `new_order`, `payment_received`, `cancel_requested`, `address_update_requested`, `order_deleted`.

### Cart

Stored in `localStorage` only (`src/lib/cart.ts`). State is shared via `CartProvider` (`src/lib/cart-context.tsx`) — call `useCart()` from there, not from any standalone hook. `CartProvider` is mounted in `layout.tsx` so all consumers share one instance. No server state. Cart quantities are validated against live Firestore stock when the checkout page loads.

### Middleware

`src/proxy.ts` exports the Next.js middleware function and `config` matcher for `/admin/*`. It currently passes through (admin auth is enforced in the layout and API routes, not the middleware edge function).

### Key shared types

`src/lib/types.ts` defines all shared interfaces:

- **Core:** `Product`, `Category`, `Order`, `OrderItem`, `ShippingAddress`, `AppUser`, `CartItem`
- **Notifications:** `Notification`, `NotificationType`
- **Union types:** `PaymentMethod` (`"vietqr" | "momo"`), `PaymentStatus` (`"pending" | "paid" | "failed"`), `OrderStatus` (`"pending" | "confirmed" | "shipping" | "delivered" | "cancelled"`)

`Order` includes `shippingFee` and `subtotal` in addition to `totalAmount`. `ShippingAddress` includes `ward` and `province` (Vietnamese address hierarchy).

### Validation

`src/lib/validation.ts` exports `shippingAddressSchema` (Zod 4) and `parseShippingErrors()`. The schema validates `name`, `phone` (regex + min-digits), `address`, `ward`, `province`. Used in both `POST /api/orders` and `PATCH /api/orders/[id]` for address updates.

### Vietnamese address data

`src/data/vn-address.json` contains province/ward hierarchy. `src/lib/vn-address.ts` exports `VnProvince`, `VnWard`, and helpers to list provinces and look up wards by province code. Used in `ShippingForm` to populate address dropdowns.

### Format utilities

`src/lib/format.ts` exports `formatPrice(amount, locale?)` and `formatDate(date, locale?)`. Both default to `"vi-VN"` locale. Pass `"en-US"` for English formatting. Currency is always VND.

### Internationalisation (i18n)

Client-side only. `src/lib/i18n/` contains:

- `translations.ts` — `Locale` (`"vi" | "en"`) and `TranslationKey` union (~140 keys)
- `en.ts` / `vi.ts` — typed `Record<TranslationKey, string>` dictionaries
- `locale-context.tsx` — `LocaleProvider` + `useLocale()` hook

`LocaleProvider` wraps the app in `layout.tsx` (same pattern as `AuthProvider`). `useLocale()` returns `{ locale, setLocale, t }` where `t(key)` looks up the active dictionary. The selected locale is persisted in `localStorage` under the key `souvenir-shop-locale` and defaults to `"vi"`. A toggle button in the Header switches languages. Pass `locale` from `useLocale()` into `formatPrice`/`formatDate` to match number/date formatting to the active language.

### Confirm dialog

`src/lib/confirm-context.tsx` provides `ConfirmProvider` + `useConfirm()`. `useConfirm()` returns an async `confirm(message)` function that resolves `true`/`false`. Wraps `ConfirmDialog` (`src/components/ui/ConfirmDialog.tsx`). Mounted in `layout.tsx`. Use instead of `window.confirm`.

### Admin UI

Pages under `src/app/admin/`:

- `page.tsx` — dashboard with KPI cards, recent orders table, top products, order status breakdown (all via client Firestore reads)
- `orders/page.tsx` — full order management
- `products/page.tsx` — product CRUD with image upload

Admin-specific components in `src/components/admin/`: `KPICards`, `RecentOrdersTable`, `TopProducts`, `StatusBreakdown`, `OrderActions`, `ProductForm`, `ImageUploader`.

## Firestore security rules

`firestore.rules` — products/categories are public read; orders are write-protected (all mutations via Admin SDK); `isAdmin` field on user documents is write-protected from clients.
